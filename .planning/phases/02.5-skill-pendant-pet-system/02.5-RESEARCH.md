## Summary

Phase 2.5 implement hệ thống trigger-based buff hoàn chỉnh. Không phải stat aggregation (D-30 đã hủy effective_stat()). Mỗi trong số 27 skills, 12 pendants, và 4 pets là một khối procedural logic riêng biệt với trigger point, rate formula, và effect handler.

Codebase hiện tại đã có sẵn phần lớn infrastructure cần thiết: FSM 7 phases, EventBus publish/subscribe, Player dataclass, Board dataclass, calc_invested_build_cost(), _toll_modifiers.py, và ConfigLoader. Công việc chủ yếu là: (1) thêm fields vào Player/Board, (2) redesign schemas, (3) tạo YAML configs, (4) viết 43 handler functions, và (5) inject hooks tại đúng điểm trong FSM.

Rủi ro kỹ thuật chính: hook injection vào FSM đang hoạt động mà không phá vỡ 241 tests hiện có. Cách tiếp cận an toàn nhất là callback injection pattern (thay vì subclassing FSM), nhất quán với cách `buy_decision_fn`, `accept_card_fn`, v.v. đã được xây dựng.

**Primary recommendation:** Implement trigger hook system bằng SkillEngine class với dict dispatch per trigger point. FSM gọi `skill_engine.fire(trigger, context)` tại mỗi hook point. Handler functions nhỏ, dễ test, và có thể được thêm/sửa độc lập.

### Pattern 1: SkillEngine với Dict Dispatch

**What:** SkillEngine nhận danh sách skills của player, expose method `fire(trigger: str, ctx: dict)`. Khi FSM đến hook point → gọi `fire()` → dispatch đến handler đúng.

**When to use:** Tất cả trigger points trong Phase 2.5.

**Example:**

```python
# Source: [ASSUMED — design pattern dựa trên codebase inspection]
class SkillEngine:
    def __init__(self, player: Player, skill_configs: dict[str, SkillEntry]):
        self.player = player
        self.configs = skill_configs

    def fire(self, trigger: str, ctx: dict) -> list[GameEvent]:
        """Fire all skill/pendant/pet handlers for given trigger point."""
        if self.player.skills_disabled_this_turn:
            return []
        events = []
        for skill_id in self.player.skills:
            cfg = self.configs.get(skill_id)
            if cfg and cfg.trigger == trigger:
                rate = self._calc_rate(cfg)
                if random.randint(0, 100) < rate:
                    handler = _SKILL_HANDLERS.get(skill_id)
                    if handler:
                        events.extend(handler(self.player, ctx))
        return events

    def _calc_rate(self, cfg: SkillEntry) -> float:
        rank = self.player.rank
        if rank == "R":
            rank = "S"  # D-03: R dùng config S
        rank_cfg = cfg.rank_config.get(rank)
        if rank_cfg is None:
            return 0.0  # D-04: không có config → không xuất hiện
        return rank_cfg.base_rate + (self.player.star - rank_cfg.min_star) * rank_cfg.chance
```

### Anti-Patterns to Avoid

- **Subclassing GameController:** Sẽ phá vỡ 241 tests. Dùng composition (inject engine).
- **Inline skill logic trong FSM:** Sẽ tạo 500-line _do_roll(). Tách ra handlers module.
- **Reload YAML mỗi lần check:** Load 1 lần vào ConfigLoader, pass dict xuống SkillEngine.
- **Dùng stat_deltas cho trigger-based effects:** D-30 đã hủy — mọi effect đều procedural.
- **if/elif chain per skill_id trong FSM:** Dict dispatch hoặc registry pattern.

## Common Pitfalls

### Pitfall 1: skills_disabled_this_turn không reset cuối lượt

**What goes wrong:** Player bị SK_LOC_XOAY disable skills. Flag không được clear → tất cả lượt sau cũng bị disable.
**Why it happens:** `skills_disabled_this_turn` và `cards_disabled_this_turn` phải reset trong `_do_end_turn()` nhưng dễ bị bỏ quên vì có nhiều fields mới trên Player.
**How to avoid:** Thêm 1 block rõ ràng "reset per-turn flags" vào `_do_end_turn()` hoặc `_do_roll()` đầu lượt. Liệt kê tất cả fields cần reset: `skills_disabled_this_turn`, `cards_disabled_this_turn`, `cam_co_decay_index`, `cam_co_current_rate`.
**Warning signs:** Test với 2 players, SK_LOC_XOAY active → verify player bị lock chỉ 1 lượt.

### Pitfall 2: bound_turns (PET_TROI_CHAN) chặn roll thay vì chặn move

**What goes wrong:** `bound_turns > 0` chỉ cần chặn di chuyển nếu tung số lẻ, không phải skip toàn bộ lượt.
**Why it happens:** Dễ implement sai logic: skip MOVE phase thay vì check dice result trong MOVE.
**How to avoid:** Check `bound_turns` sau khi roll: nếu lẻ → `player.bound_turns -= 1`, skip MOVE. Nếu chẵn → tiếp tục bình thường (và decrement). Decrement phải xảy ra dù chẵn hay lẻ.
**Warning signs:** bound_turns > 0 nhưng player vẫn move → logic sai vị trí check.

### Pitfall 3: illusion_position và stop_sign_position không được reset khi Board.reset_for_new_game()

**What goes wrong:** Ảo ảnh hoặc biển cấm từ ván trước còn tồn tại sang ván mới.
**Why it happens:** `Board.reset_for_new_game()` hiện chưa reset các fields mới (chưa được thêm vào).
**How to avoid:** Khi thêm `illusion_position` và `stop_sign_position` vào Board, đồng thời cập nhật `reset_for_new_game()` để set cả 2 về None.
**Warning signs:** Tests chạy reset_for_new_game() nhưng illusion_position vẫn != None.

### Pitfall 4: pet_stamina depletion timing

**What goes wrong:** Pet kích hoạt trước khi decrement stamina, hoặc không decrement khi active.
**Why it happens:** D-40 nói "mỗi lần skill active → tiêu 1 thể lực" — timing phải: check stamina > 0, THEN check rate, THEN if active → decrement stamina AND apply effect.
**How to avoid:** Pattern cố định:
```python
if player.pet_stamina > 0 and random.randint(0, 100) < rate:
    player.pet_stamina -= 1
    apply_pet_effect(...)
```
**Warning signs:** PET_THIEN_THAN (max_stamina=1) kích hoạt 2 lần trong 1 ván.

### Pitfall 5: Pendant dual-rate serialization

**What goes wrong:** PT_XICH_NGOC có Rate1 (tỷ lệ ra tù) và Rate2 (điều khiển xúc xắc) nhưng chúng map tới 2 triggers khác nhau (ON_PRISON_ESCAPE_CHECK và ON_DKXX_CHECK).
**Why it happens:** Nếu schema chỉ có 1 `rank_rates` dict, sẽ không rõ rate nào dùng cho trigger nào.
**How to avoid:** PendantEntry cần `rank_rates` (cho trigger chính) và `rank_rates_2` optional (cho trigger phụ). Hoặc dùng nested dict `{trigger_name: {rank: rate}}`.
**Warning signs:** PT_SIEU_TAXI Rate1 (di chuyển khi tới travel) bị áp dụng sai trigger.

### Pitfall 6: SK_JOKER TH2 (joker_pending) không được xử lý đầu lượt

**What goes wrong:** joker_pending = True được set khi player bị đẩy vào tù ngoài lượt, nhưng không có code kiểm tra flag này đầu lượt tiếp theo.
**Why it happens:** Flag mới trên Player, cần hook vào đầu _do_roll() để check và process.
**How to avoid:** Đầu `_do_roll()`: if `player.joker_pending` → move to nearest unowned CITY, grant extra roll, clear flag.
**Warning signs:** joker_pending = True sau end_turn nhưng không bao giờ xử lý.

### Pitfall 7: Config schema không backward-compatible

**What goes wrong:** SkillEntry hiện tại có `extra="forbid"` và `stat_deltas: dict[str, float]`. Khi thêm fields mới, validation fails trên skills.yaml rỗng hiện tại.
**Why it happens:** `skills.yaml` hiện tại chỉ có `skills: []` — load thành công vì list rỗng. Nhưng sau khi thêm entries mới, fields cũ như `stat_deltas` không còn phù hợp.
**How to avoid:** Redesign schema hoàn toàn (không backward-compat), cập nhật skills.yaml đồng thời. Test `ConfigLoader.load_all()` sau mỗi thay đổi.
**Warning signs:** Pydantic ValidationError khi load skills.yaml sau khi thêm dữ liệu.

## Open Questions (RESOLVED)

1. **D-17: 1 file per skill vs monolithic YAML** — RESOLVED
   - Decision: Use monolithic `ctp/config/skills.yaml` instead of per-file `ctp/config/skills/` directory.
   - Rationale: ConfigLoader already reads `skills.yaml` monolithic. Refactoring to glob-load 27 separate files adds loader complexity with zero functional benefit for this phase. Single load point is simpler to validate (one Pydantic parse) and debug. D-17 locked decision is deliberately overridden here because the monolithic approach was already established in the codebase and produces identical runtime behavior.
   - Override documentation: This is an intentional deviation from D-17 ("1 file per skill trong ctp/config/skills/"). The user's intent (configurable skill pool via YAML) is fully preserved; only the file organization differs. If per-file loading is needed later, it can be added as a ConfigLoader enhancement without changing any skill logic.

2. **`consecutive_doubles` vs `_doubles_streak`** — RESOLVED
   - Decision: Pass `controller` in the skill context dict (`ctx["controller"]`). SK_HQXX handler resets `ctx["controller"]._doubles_streak = 0` directly.
   - Rationale: `Player.consecutive_doubles` (D-15) tracks the player-visible doubles count. `GameController._doubles_streak` is the FSM-internal field that triggers triple-doubles prison. SK_HQXX needs to reset the FSM field to prevent triple-doubles prison, so it must access the controller. The context dict already includes `"controller": self` per the `_skill_ctx()` helper in Plan 02 Task 2.

3. **ON_OPPONENT_COMPLETE_2_COLOR_PAIRS check** — RESOLVED
   - Decision: After every ACQUIRE and UPGRADE phase in FSM, loop ALL non-bankrupt players. For each player, count complete color pairs via `_count_complete_color_pairs(player)`. If any player has 2+ complete color pairs, fire `ON_OPPONENT_COMPLETE_2_COLOR_PAIRS` for all OTHER non-bankrupt players (so PET_XI_CHO owners can react).
   - Rationale: The trigger is about any opponent reaching a threshold, not just the current player. Checking all players ensures the pet fires regardless of who caused the pair completion. The `_count_complete_color_pairs()` helper is defined in Plan 02 Task 2 and uses `board.get_color_group_positions()` to count owned complete groups.
