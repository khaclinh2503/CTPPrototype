### Root Cause (Unchanged from Previous Verification)

The GAP-SUMMARY (`02.1-GAP-SUMMARY.md`) states "Files modified: 0 (implementation pre-existed in commit d24ec3b)." This is **false**:

1. `git grep virus_turns` across the entire repo returns 0 results in any production source file

3. The GAP-PLAN execution committed no code changes — it only created the SUMMARY file
4. Commit d24ec3b (feat 02.1-03 FSM integration) never modified `ctp/core/models.py`

The virus mechanic exists as tile-level debuff (`tile.toll_debuff_turns` + `tile.toll_debuff_rate` on the Tile dataclass). This was the original Phase 02.1 implementation before the D-11 spec was read as requiring player-level semantics. The GAP plan correctly identified the discrepancy but the GAP executor incorrectly concluded the player-level version pre-existed.

**Gap 2: Virus mechanic is entirely tile-level — four files need updating**
The current implementation stores virus state on individual `Tile` objects (`toll_debuff_turns`, `toll_debuff_rate`). The D-11/D-22/D-44/D-46 spec requires player-level semantics (one `virus_turns` field on `Player` covers all owned tiles, cleared on first visit by any visitor):
- `fortune.py` `_apply_color_pair_debuff()`: sets per-tile debuff instead of `opponent.virus_turns = 3`
- `_toll_modifiers.py` Step 1 (line 45): checks `tile.toll_debuff_turns > 0` instead of `owner.virus_turns > 0`
- `fsm.py` `_do_end_turn()` (lines 835-839): decrements tiles, no player loop at all
- `tests/test_cards.py`: 4 virus tests assert tile-level state — must be updated to player-level assertions
