## Summary

Phase 02.1.1 adds a minimal Pygame window to visualise the headless game already running. The core challenge is the threading model: `GameController.step()` must run in a background thread (so the render loop stays responsive) while a `threading.Lock` protects the shared UI state dict that EventBus handlers write into and the render loop reads from.

The board is a 32-tile diamond. The geometry is straightforward: the board fits inside a rotated square with a configurable radius R. Each of the 4 sides has 8 tiles evenly spaced between corner positions (1, 9, 17, 25). Each tile is rendered as a small rhombus (`pygame.draw.polygon`) centred at its computed position. Player tokens are short strings rendered with `pygame.font.Font` directly on top of tile centres, offset slightly when multiple players share a tile.

The info panel (right ~1/3) and event log (below info panel) receive all their data from the UI state dict. The speed controller lives entirely in the background thread — a `threading.Event` acts as the pause signal and `time.sleep()` provides the delay between steps. Pygame 2.6.1 is available on the machine (just verified via `pip install`); SDL requirement that display init and event processing happen on the main thread is the canonical constraint that drives the two-thread design.

**Primary recommendation:** Use `threading.Event` (pause signal) + `threading.Lock` (shared state) + `time.sleep()` (delay) in the background thread; Pygame event loop + render on main thread. Subscribe EventBus handlers before starting the background thread to avoid race conditions at game start.

### Pattern 1: Two-Thread Design

**What:** Main thread owns Pygame (SDL requirement). Background thread runs `GameController.step()` in a loop with configurable `time.sleep()`.

**When to use:** Any case where game logic must not block the render loop. SDL requires all display operations on the same OS thread that called `pygame.init()`.

**Example:**
```python
# Source: pygame documentation + Python threading stdlib
import threading, time

class SpeedController:
    DELAYS = {"pause": None, "1x": 0.8, "5x": 0.16, "max": 0.0}

    def __init__(self, controller):
        self._controller = controller
        self._speed = "1x"
        self._pause_event = threading.Event()
        self._pause_event.set()   # start unpaused
        self._lock = None         # injected by GameView
        self._thread = threading.Thread(target=self._run, daemon=True)

    def _run(self):
        while not self._controller.is_game_over():
            self._pause_event.wait()           # blocks when paused
            self._controller.step()
            delay = self.DELAYS.get(self._speed, 0.8)
            if delay:
                time.sleep(delay)

    def toggle_pause(self):
        if self._pause_event.is_set():
            self._pause_event.clear()
            self._speed = "pause"
        else:
            self._pause_event.set()
            self._speed = "1x"   # resume at 1x

    def set_speed(self, speed: str):
        self._speed = speed
        if speed != "pause":
            self._pause_event.set()
        else:
            self._pause_event.clear()

    def start(self):
        self._thread.start()
```

### Pattern 2: EventBus Subscription for UI State

**What:** Subscribe EventBus handlers before the background thread starts. Each handler acquires the lock, updates the UI state dict, appends to the event log deque.

**When to use:** Any time UI must react to game events without polling or direct object access.

**Example:**
```python
# Source: ctp/core/events.py EventBus.subscribe() pattern
import threading
from collections import deque
from ctp.core.events import EventType

class GameView:
    def __init__(self, event_bus, players):
        self._lock = threading.Lock()
        # Lightweight UI state — only what the render loop needs
        self._ui_state = {
            p.player_id: {"cash": p.cash, "total_assets": p.cash, "position": p.position}
            for p in players
        }
        self._board_ownership = {}   # {position: owner_id}
        self._event_log = deque(maxlen=10)
        self._active_player_id = players[0].player_id

        # Subscribe before starting background thread
        for et in [EventType.PLAYER_MOVE, EventType.RENT_PAID,
                   EventType.PROPERTY_PURCHASED, EventType.PROPERTY_ACQUIRED,
                   EventType.TURN_STARTED, EventType.GAME_ENDED,
                   EventType.PLAYER_BANKRUPT]:
            event_bus.subscribe(et, self._on_event)

    def _on_event(self, event):
        with self._lock:
            # update _ui_state, _board_ownership, _event_log
            ...

    def render(self, screen):
        with self._lock:
            state = dict(self._ui_state)  # shallow copy under lock
            log = list(self._event_log)
            ownership = dict(self._board_ownership)
        # render from copied data — no lock held during drawing
        ...
```

### Pattern 3: Diamond Geometry

**What:** Compute tile centre positions analytically. Board fits in a diamond with half-radius R. 4 sides of 8 tiles each; corners at positions 1, 9, 17, 25.

**When to use:** Board rendering. Keeps coordinate calculation in one function with no hardcoded per-tile positions.

**Example:**
```python
# Source: verified by coordinate math (2026-04-04)
import math

def tile_center(pos: int, cx: float, cy: float, R: float) -> tuple[float, float]:
    """Compute screen centre for tile at board position pos (1-32).
    
    Diamond corners: pos 1 = bottom, pos 9 = right, pos 17 = top, pos 25 = left.
    Clockwise traversal. 8 tiles per side, corners at multiples of 8 + 1.
    """
    i = (pos - 1) % 8      # 0-indexed position within side
    side = (pos - 1) // 8  # 0=bottom-right, 1=right-top, 2=top-left, 3=left-bottom
    t = i / 8              # interpolation factor along side

    if side == 0:          # bottom (pos 1) → right (pos 9)
        return (cx + t * R, cy + R - t * R)
    elif side == 1:        # right (pos 9) → top (pos 17)
        return (cx + R - t * R, cy - t * R)
    elif side == 2:        # top (pos 17) → left (pos 25)
        return (cx - t * R, cy - R + t * R)
    else:                  # left (pos 25) → bottom (pos 1)
        return (cx - R + t * R, cy + t * R)

def tile_polygon(cx: float, cy: float, s: float) -> list[tuple[float, float]]:
    """4-point rhombus centred at (cx, cy) with half-size s."""
    return [(cx, cy - s), (cx + s, cy), (cx, cy + s), (cx - s, cy)]
```

**Verified coordinates for R=280, cx=427, cy=360 (1280×720 window, board in left 2/3):**
- pos 1 (bottom): (427, 640)
- pos 9 (right): (707, 360)
- pos 17 (top): (427, 80)
- pos 25 (left): (147, 360)
- Center spacing per tile: ~49.5 px → tile half-size s ≈ 24 px

### Anti-Patterns to Avoid

- **Reading Player/Board objects in render loop without lock:** Player dataclass is mutated by the background thread. Reading `.cash` or `.position` outside a lock causes data races. Use the UI state dict copy instead.
- **Calling `pygame.display` functions from background thread:** SDL requires display operations on the main thread. All drawing must stay in the render loop.
- **Subscribing EventBus after thread starts:** If the background thread fires events before subscriptions are registered, early events are silently dropped from the log.
- **Using `pygame.event.pump()` instead of `pygame.event.get()`:** On some platforms, skipping `event.get()` causes the OS to report the window as unresponsive. Always drain the event queue every frame.
- **Not using `daemon=True` on background thread:** If the user closes the window without the game ending, the process hangs waiting for the background thread. `daemon=True` ensures the thread dies with the main process.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Event log ring buffer | Manual list with pop(0) | `collections.deque(maxlen=10)` | Thread-safe append, O(1) pop-left, auto-evicts oldest |
| Pause/resume signalling | Boolean flag + spin-wait | `threading.Event` | `wait()` releases GIL during sleep; boolean polling wastes CPU |
| Frame rate limiting | Manual `time.sleep()` in render loop | `pygame.time.Clock.tick(60)` | Accounts for frame render time; prevents spiral-of-death |
| Font rendering | Loading external TTF files | `pygame.font.SysFont(None, size)` | Uses OS default font; no external assets; decision locked (D-discretion) |

**Key insight:** The threading primitives in Python stdlib are production-quality. The only custom code needed is the wiring between EventBus handlers and the UI state dict.

## Common Pitfalls

### Pitfall 1: GIL Does Not Protect Pygame Drawing
**What goes wrong:** Developer assumes Python GIL prevents rendering crashes when background thread calls any Pygame function.
**Why it happens:** GIL only protects Python bytecode interleaving; SDL C library has its own internal state that is not GIL-protected.
**How to avoid:** Never call any `pygame.*` function from the background thread. All rendering, font creation, surface operations must be on the main thread.
**Warning signs:** Sporadic crashes on Windows with no traceback (SDL assertion failures).

### Pitfall 2: EventBus Handlers Running in Background Thread Context
**What goes wrong:** EventBus.publish() is called synchronously inside `controller.step()` in the background thread. That means all EventBus handlers registered by the UI also run in the background thread.
**Why it happens:** EventBus dispatches handlers inline at publish time (see `ctp/core/events.py` line 144–146).
**How to avoid:** All EventBus handlers that touch the UI state dict MUST acquire `_lock` before writing. The render loop MUST acquire `_lock` before reading. This is the critical invariant for the whole design.
**Warning signs:** Intermittent `KeyError` or incorrect player positions in the display.

### Pitfall 3: Stale State on Game End
**What goes wrong:** Background thread calls `is_game_over()` and exits its loop. The render loop never sees the GAME_ENDED event because the EventBus handler was never registered for it.
**Why it happens:** GAME_ENDED is published inside `_do_end_turn()` which is called in the background thread, after the is_game_over() check passes on the next call.
**How to avoid:** Subscribe to `EventType.GAME_ENDED` in the UI. In the handler, set a flag (under lock) so the render loop can display "GAME OVER" state. Do not rely on polling `is_game_over()` from the render loop.
**Warning signs:** Window goes blank or freezes when game ends normally.

### Pitfall 4: Total Assets Calculation Requires Board Access
**What goes wrong:** D-07 requires showing `total_assets = cash + Σ(build_cost per owned tile)`. Calculating this on every render frame means reading Board objects during the render loop.
**Why it happens:** Build costs are in `board.get_land_config(tile.opt)["building"]`.
**How to avoid:** Calculate total_assets inside the EventBus handlers (when `PROPERTY_PURCHASED`, `PROPERTY_ACQUIRED`, `PROPERTY_UPGRADED`, `PROPERTY_SOLD`, `TURN_STARTED`, or `TURN_ENDED` fires) — at that point the background thread has the Board under its natural execution context. Store the pre-computed value in the UI state dict. The render loop only reads the already-computed number.
**Warning signs:** Incorrect total_assets display or crashes when render loop reads Board.

### Pitfall 5: requirements.txt Does Not Include pygame
**What goes wrong:** `pygame` is not in `requirements.txt`, so a fresh clone fails to run the UI.
**Why it happens:** It was just installed manually; not committed to requirements.
**How to avoid:** Add `pygame>=2.6.1` to `requirements.txt` as part of the Wave 0 setup task.
**Warning signs:** `ModuleNotFoundError: No module named 'pygame'` on a clean install.
